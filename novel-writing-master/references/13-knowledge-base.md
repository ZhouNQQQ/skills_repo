# 小说知识库管理

借鉴软件工程中的知识管理方法（CONTEXT.md + ADR），建立小说创作的"共享语言+决策记录+剧情索引"系统。解决长篇幅创作的核心问题：**写到后面忘了前面**。

## 核心原则

1. **知识库是唯一真相源**：所有设定、决策、剧情状态以文件为准，不以对话历史为准。AI 重启后对话历史消失，知识库文件不会。
2. **先读后写**：每次创作会话开始前，必须先读取知识库了解当前状态。
3. **边写边记**：产生新设定、新决策、新伏笔时，**立即更新**对应文件，不批量、不拖延。
4. **决策即记录**：任何对设定、剧情、角色的重大变更，必须同时记录决策原因。
5. **知识库跟着项目走**：知识库文件存放在你的小说项目目录下，技能只负责读写。

---

## 知识库放在哪里

知识库文件放在**你的小说项目目录下**，和章节文件在一起。这是默认方案。

### 默认方案：本地文件

```
my-novel/                    # 你的小说项目文件夹
├── chapters/
│   ├── 001.md
│   └── 002.md
├── novel-bible.md           # 总览索引 + 雪花步骤追踪
├── outline.md               # 雪花大纲：一句话→五句段→详细大纲→场景清单
├── worldbuilding.md         # 世界观
├── characters.md            # 雪花人物卡 + 完整小传 + 运行时档案
├── plot.md                  # 剧情
├── chapter-index.md         # 章节索引
├── foreshadowing-log.md     # 伏笔
├── consistency-log.md       # 一致性
└── decisions/
    ├── D0001-power-system.md
    └── D0002-character-arc.md
```

### 进阶方案：GitHub 仓库（推荐）

把知识库存成 GitHub 仓库，获得**版本历史、多设备同步、团队协作、回滚能力**。

#### 需要准备什么

1. **GitHub 账号**（免费）
2. **Personal Access Token**（经典版，勾选 `repo` 权限）
3. **仓库**（公开或私有均可）

#### 仓库结构

```
my-novel-kb/                 # 知识库专用仓库（可与正文仓库分离）
├── README.md               # 项目说明（可选）
├── novel-bible.md          # 总览索引 + 雪花步骤追踪
├── outline.md              # 雪花大纲：一句话→五句段→详细大纲→场景清单
├── worldbuilding.md
├── characters.md           # 雪花人物卡 + 完整小传 + 运行时档案
├── plot.md
├── chapter-index.md
├── foreshadowing-log.md
├── consistency-log.md
└── decisions/
    ├── D0001-xxx.md
    └── D0002-xxx.md
```

#### Token 配置方式

用户需在对话开始时提供 Token，或预先配置到环境变量：

```bash
export GITHUB_TOKEN="ghp_xxxxxxxxxxxx"
export KB_REPO="owner/repo-name"      # 如：zhangsan/my-xianxia-kb
export KB_BRANCH="main"              # 默认分支
```

**安全提示**：Token 只用于当前会话，不存储到 skill 文件中。

#### 操作命令模板

配置了 Token 后，Agent 可以通过以下方式操作：

**读取知识库（会话开始时）**
```bash
# 下载单个文件
curl -s -H "Authorization: token $GITHUB_TOKEN" \
  "https://raw.githubusercontent.com/$KB_REPO/$KB_BRANCH/novel-bible.md"

# 下载全部文件（通过 GitHub API 列目录）
gh api repos/$KB_REPO/contents/ --jq '.[].name'
```

**写入知识库（会话结束时）**
```bash
# 更新文件（需要获取文件 SHA，然后提交）
# 这是一个两步操作：1.获取SHA 2.提交更新
curl -s -H "Authorization: token $GITHUB_TOKEN" \
  -X PUT "https://api.github.com/repos/$KB_REPO/contents/novel-bible.md" \
  -d '{
    "message": "update: chapter 6 summary and foreshadowing",
    "content": "'$(base64 novel-bible.md)'",
    "sha": "'$(获取到的SHA)'"
  }'
```

**简化方案：使用 `gh` CLI（如果环境中已安装）**
```bash
# 读取
gh api repos/$KB_REPO/contents/novel-bible.md --jq '.content' | base64 -d

# 写入（先生成 base64 内容）
CONTENT=$(base64 novel-bible.md | tr -d '\n')
gh api repos/$KB_REPO/contents/novel-bible.md \
  -X PUT \
  -F message="update from session" \
  -F content="$CONTENT" \
  -F sha="$(获取到的SHA)"
```

#### 工作流示例（GitHub 版）

```
用户：继续写第6章
→ 询问知识库位置
   → 如果是 GitHub：询问 owner/repo 和 branch
   → 如果是本地：询问本地路径
→ 读取知识库（从 GitHub 下载或本地读取）
→ 写第6章
→ 更新知识库文件
→ 同步回 GitHub（提交变更）或提示用户保存本地文件
```

#### 提交信息规范

```
update: chapter N summary
fix: character power consistency (C0003)
add: foreshadowing F0005 for later reveal
decision: D0006 why villain chooses X
```

### 其他可选方案

| 方案 | 优点 | 缺点 |
|------|------|------|
| **本地文件** | 最简单，零配置 | 无版本历史，换设备麻烦 |
| **GitHub 仓库** | 版本控制、多设备、协作 | 需要 Token，有学习成本 |
| **Obsidian + Git** | 可视化、图谱、本地优先 | 配置复杂 |
| **坚果云/iCloud** | 自动同步、零配置 | 无版本历史，冲突难处理 |

**推荐**：先用本地文件跑起来，写够 5 章后再迁移到 GitHub。

## 六文件+一目录的分层设计

把以前"塞在一个 novel-bible.md 里"的大杂烩拆成六份，每份职责单一、更新频率独立。

| 文件 | 职责 | 更新频率 | 雪花关联 | 类比（软件工程） |
|------|------|----------|----------|-----------------|
| **novel-bible.md** | 导航总览：书名、当前进度、文件清单、雪花步骤追踪 | 极低 | 步骤1（故事核） | README |
| **outline.md** | **雪花核心**：一句话→五句段→详细大纲→场景清单 | 中 | 步骤1/2/4/6 | 架构蓝图 |
| **worldbuilding.md** | 世界观冻结区+扩展区 | 中 | - | CONTEXT |
| **characters.md** | **雪花人物卡+完整小传**+运行时档案 | 高 | 步骤3/5 | 对象定义 |
| **plot.md** | 核心矛盾+已揭示+待揭示信息 | 中 | - | 架构图 |
| **chapter-index.md** | 章节索引：每章摘要、出场角色、情绪 | 最高 | 步骤7（对照场景写作） | CHANGELOG |
| **foreshadowing-log.md** | 伏笔追踪 | 高 | - | Issue Tracker |
| **consistency-log.md** | 一致性检查 | 中 | - | Bug Tracker |
| **decisions/** | 决策记录 | 低 | - | ADR |

### 拆分原则

**拆分的标准不是"内容多少"，而是"更新频率"**。更新频率不同的内容不应该放在同一个文件里——否则你每次更新高频内容都要重写整个文件。

## 文件一：novel-bible.md（总览索引）

**这是知识库的"首页"**。保持极简，只放导航信息。其他文件是它的子页面。

```markdown
# 《书名》小说圣经

最后更新：YYYY-MM-DD
当前进度：第 X 章 / 计划共 Y 章 / 当前卷：Z
作者：...

## 一句话故事核（雪花步骤1）
```
（待填写，15字以内：谁+做什么+代价/阻碍）
```
> 故事核一旦确定，全书不得偏离。修改需在 decisions/ 记录原因。

## 雪花步骤完成追踪

- [ ] 步骤1：一句话故事核
- [ ] 步骤2：五句段概括 → [outline.md](outline.md)
- [ ] 步骤3：雪花人物卡 → [characters.md](characters.md)
- [ ] 步骤4：扩情节 → [outline.md](outline.md)
- [ ] 步骤5：扩人物 → [characters.md](characters.md)
- [ ] 步骤6：详细大纲+场景清单 → [outline.md](outline.md)
- [ ] 步骤7：写正文 → [chapter-index.md](chapter-index.md)

> 网文适配：步骤1-3动笔前完成；步骤4-5可与连载同步；步骤6每卷开始前做一次。

## 知识库文件导航

| 文件 | 内容 | 上次更新 |
|------|------|----------|
| outline.md | 雪花大纲：一句话→五句段→详细大纲→场景清单 | YYYY-MM-DD |
| worldbuilding.md | 世界观规则、力量体系、势力、地理 | YYYY-MM-DD |
| characters.md | 雪花人物卡+完整小传+运行时档案 | YYYY-MM-DD |
| plot.md | 核心矛盾、已揭示信息、待揭示信息 | YYYY-MM-DD |
| chapter-index.md | 章节摘要索引（对照场景清单写作） | YYYY-MM-DD |
| foreshadowing-log.md | 伏笔追踪 | YYYY-MM-DD |
| consistency-log.md | 一致性检查 | YYYY-MM-DD |
| decisions/ | 重要创作决策记录 | N 条 |

## 快速状态栏

- **当前主角目标**：...
- **当前核心威胁**：...
- **最近埋设伏笔**：...（F编号）
- **最近回收伏笔**：...（F编号）
- **待处理一致性问题**：...（C编号，如有）
```

**规则**：这个文件不要超过 50 行。它的唯一职责是让你（和 AI）一眼知道知识库的结构和当前状态。

## 目录：decisions/（创作决策记录）

借鉴软件工程的 ADR（Architecture Decision Record）。每条重要决策独立一个文件。

### 何时记录决策

**三个条件全部满足**时才创建：
1. **难以逆转**：后期更改会产生较大连锁修改成本
2. **没有上下文会令人困惑**：未来的你会问"为什么这样设计？"
3. **真实权衡的结果**：当时有其他可选方案，你经过思考后选择了这个

### 格式

文件名：`D0001-{简短描述}.md`，编号递增。

```markdown
# D0001：主角金手指设定为"机械之心"

主角的金手指设定为"机械之心"——吞噬机械获得技能/扫描蓝图制造/意念驱动。

原因是：该能力与末世重建文明的主线强绑定，且"机械思维侵蚀人性"的副作用
能持续制造内在冲突。考虑过"元素操控"和"预知未来"，但前者与重建主题关联弱，
后者容易破坏悬念。
```

就这么多。1-3句话：**决策是什么**、**为什么**、**考虑过什么替代方案**。

### 什么值得记录 / 什么不需要

| 值得记录 | 不需要记录 |
|----------|-----------|
| 力量体系核心规则 | "这是一个修真世界"（显然） |
| 重要角色的核心动机/秘密 | "本章用暴雨天气营造氛围"（临时） |
| 关键剧情转折的设计理由 | 没有替代方案时的选择 |
| 设定变更及其原因 | |
| 题材/风格/基调的确定或变更 | |

## 更新时机与操作流程

### 会话开始（读取）

```
用户：继续写第6章
→ 询问知识库位置（如不知道，建议放在项目目录）
→ 读取 novel-bible.md（了解全局状态）
→ 读取 characters.md（了解角色当前状态）
→ 读取 chapter-index.md（回顾前5章剧情）
→ 读取 foreshadowing-log.md（查看未回收伏笔，可能用作素材）
→ 读取 consistency-log.md（确认没有待处理冲突）
→ 开始创作第6章
```

**铁律**：在没有读取知识库之前，不产出任何正文创作内容。

### 创作过程中（实时更新）

| 事件 | 更新文件 | 具体操作 |
|------|----------|----------|
| 产生了新的世界观设定 | worldbuilding.md | 添加到"冻结区"或"扩展区" |
| 角色能力/性格发生变化 | characters.md | 更新角色档案中的当前状态 |
| 埋设了伏笔 | foreshadowing-log.md | 添加到"活跃伏笔"，分配F编号 |
| 回收了伏笔 | foreshadowing-log.md | 移到"已回收伏笔" |
| 揭示了重要信息 | plot.md + worldbuilding.md | 更新"已揭示清单" |
| 完成了第N章 | chapter-index.md | 添加第N章摘要和速查行 |
| 做出了重要剧情决策 | decisions/ | 新建 D000N 文件 |
| 发现设定冲突 | consistency-log.md | 添加到"待处理" |
| 修复了设定冲突 | consistency-log.md | 移到"已修复" |

**规则**：边写边记，不要等一章写完再统一更新。想到就记，防止遗忘。

### 会话结束（同步）

```
→ 总结本次会话的所有变更
→ 向用户展示变更摘要：
   "本次更新了：
    - worldbuilding.md：新增规则X（冻结区）
    - characters.md：主角能力提升至Y级
    - chapter-index.md：添加第6章摘要
    - foreshadowing-log.md：埋设伏笔F04，回收F02
    - decisions/：新建 D0003（关于Z的决策）"
→ 提醒用户保存/提交知识库文件
→ 高亮提醒：如有未处理的 consistency 问题
```

## 知识库与创作场景的整合

### 场景一：启动新项目

```
用户：我想写一本末世修仙小说
→ 询问项目目录和知识库位置
→ 创建初始知识库文件（空模板）
→ 填充 novel-bible.md（书名、进度0）
→ 填充 worldbuilding.md（基础设定）
→ 填充 characters.md（主角初始人设）
→ 填充 plot.md（核心矛盾框架）
→ 开始创作，边写边更新
```

### 场景二：继续已有项目（标准流程）

```
用户：继续写第6章
→ 读取 novel-bible.md（全局状态）
→ 读取 characters.md（角色当前状态）
→ 读取 chapter-index.md（前5章剧情）
→ 读取 foreshadowing-log.md（找可用素材）
→ 写第6章
→ 更新 chapter-index.md（添加第6章）
→ 更新 foreshadowing-log.md（如有伏笔操作）
→ 更新 characters.md（如有角色变化）
→ 向用户展示更新摘要
```

### 场景三：卡文急救

```
用户：卡文了，不知道第20章写什么
→ 读取 foreshadowing-log.md → 找"活跃伏笔"
→ 选择一个未回收且适合本章的伏笔
→ 读取 worldbuilding.md → 检查扩展区是否有预留方向
→ 读取 plot.md → 查看"待揭示信息清单"
→ 基于以上素材设计第20章核心事件
→ 写完后更新所有受影响文件
```

### 场景四：一致性检查

```
用户：帮我检查一下设定有没有bug
→ 对比 worldbuilding.md 冻结区 vs chapter-index.md 实际剧情
→ 对比 characters.md 人设 vs 各章角色行为
→ 逐项检查 consistency-log.md 中的规则清单
→ 发现冲突 → 记录到 consistency-log.md + 提出修复方案
```

### 场景五：角色行为异常审查

```
用户：帮我看看第15章主角的行为有没有崩人设
→ 读取 characters.md 中主角档案（核心性格、能力边界、当前状态）
→ 读取 chapter-index.md 第15章摘要
→ 用"角色过滤器提问法"逐项审查：
  1. 以这个角色的性格，他会这样做吗？
  2. 能力是否在边界内？
  3. 情绪反应是否与触发事件分量匹配？
  4. 人际关系是否与当前层级一致？
→ 发现问题 → 提出修正方案 + 更新 consistency-log.md
```

## 输出标准

每次向用户展示知识库相关操作时，必须包含：
1. **变更摘要**：本次会话更新了哪些文件、哪些条目
2. **关键提醒**：是否有新产生的伏笔需要在后续回收、是否有待处理的一致性问题
3. **下一步建议**：基于知识库当前状态，建议下一步写什么
