# 技能最佳实践参考

## 描述要求（深入）

`description` 字段是**代理决定是否加载技能时唯一能看到的内容**。

### 优秀描述的组成

```
[动作动词] [具体能力]。Use when [触发条件]。
```

**组成部分：**
1. **动作动词**：技能做什么（提取、分析、生成等）
2. **具体能力**：是什么让这个技能独特
3. **触发条件**：何时使用此技能（具体短语、上下文）

### 示例

**优秀：**
```
使用模式匹配和 NLP 从非结构化文本中提取结构化数据。Use when 用户提供原始文本并要求提取实体、关键词或结构化信息。
```

**良好：**
```
根据代码分析生成测试用例。Use when 用户提到测试、测试覆盖率或要求编写测试。
```

**差：**
```
帮助进行测试。
```

**为什么差：** 太模糊、没有触发条件、无法与其他技能区分。

### 常见错误

1. **缺少触发子句**："分析代码中的 bug" → 添加 "Use when 用户报告 bug"
2. **太通用**："处理数据" → 要具体："提取、转换和验证表格数据"
3. **没有动作动词**："API 技能" → "使用重试逻辑和错误处理管理 API 请求"

### 长度指南

- **最小**：30 个字符（足以具体）
- **最大**：1024 个字符
- **最佳**：100-200 个字符（简洁但完整）

## 内容拆分策略

### 何时拆分

1. **SKILL.md 超过 100 行**
2. **有不同内容领域**（例如：金融与销售工作流）
3. **高级功能在正常使用中很少需要**
4. **会使主文件臃肿的示例和参考材料**

### 拆分模式

#### 模式 1：参考文件

将详细文档移动到 references/：

```
SKILL.md                    references/best-practices.md
├── 快速开始                ├── 技术细节
├── 工作流程                ├── 设计模式
└── 常见示例                └── 故障排除
```

#### 模式 2：示例文件

适用于有很多用例的技能：

```
SKILL.md                    EXAMPLES.md
├── 概述                    ├── 示例 1：基本用法
├── 触发条件                ├── 示例 2：高级配置
└── 工作流程                └── 示例 3：边缘情况
```

#### 模式 3：多主题拆分

适用于 reference 内容庞大的技能：

```
references/
├── severity-definitions.md     # 问题严重程度定义
├── time-sensitive-patterns.md  # 时间敏感内容识别
├── best-practices.md           # 通用最佳实践
└── review-checklist.md         # 完整审查清单
```

### 内容比例指南

| SKILL.md 行数 | 拆分方法 |
|---------------|----------|
| < 50 | 无需拆分 |
| 50-100 | 考虑拆分 |
| 100-200 | 必须拆分 |
| > 200 | 建议多个文件 |

## 脚本创建指南

### 何时使用脚本

- 操作是确定性的（相同输入总是产生相同输出）
- 代码模式在多个用例中重复
- 错误处理需要明确和一致
- 性能至关重要（编译脚本比生成代码更快）

### 脚本类型

#### 验证脚本
```javascript
// 验证技能结构
function validateStructure(skillPath) {
  const required = ['SKILL.md'];
  const missing = required.filter(f => !exists(path.join(skillPath, f)));
  return { valid: missing.length === 0, missing };
}
```

#### 格式化脚本
```javascript
// 格式化描述为标准格式
function formatDescription(desc) {
  if (!desc.includes('Use when')) {
    return desc + ' Use when [触发条件]。';
  }
  return desc;
}
```

### 脚本最佳实践

1. **快速失败并有清晰错误**：`if (!input) throw new Error('需要输入')`
2. **返回结构化结果**：`{ success: true, data: ..., errors: [] }`
3. **支持 dry-run 模式**：显示将要更改的内容而不修改文件
4. **用于调试的日志**：使用 `[DEBUG-xxx]` 前缀用于临时日志

## 反馈闭环设计

每个审查流程必须包含验证步骤：

1. **修复前**：记录问题清单和基线状态
2. **修复后**：重新分析同一检查点
3. **对比**：修复前后差异
4. **回归确认**：未引入新问题

## 工作流程图示

### 分析工作流程

```
读取 SKILL.md → 提取 YAML frontmatter → 检查结构/描述/内容 → 识别问题并评分 → 生成报告和建议
```

### 改进工作流程

```
分析技能 → 生成改进 → 向用户展示之前/之后对比 → 在确认后应用 → 重新分析验证效果
```
