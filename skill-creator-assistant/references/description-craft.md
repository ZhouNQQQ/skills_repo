# 编写技能描述

`description` 字段是你技能中最重要的部分。它是 **Agent 决定是否加载你技能的唯一信息**。

---

## 描述的作用

当用户发出请求时，Agent 扫描所有已安装技能的描述。它仅根据描述选择最相关的技能。SKILL.md 的正文在技能选择期间不会被考虑。

**这意味着**：描述不好的优秀技能永远不会被使用。好的描述是技能所有用量的入口。

---

## 格式

```
{能力陈述}。当 {特定触发条件} 时使用。
```

- **能力**：技能实现什么（1 句话）
- **触发条件**：何时激活（关键词、上下文、用户短语）

---

## 约束

- 最多 **1024 个字符**
- 用**第三人称**写作（Agent 为自己阅读）
- 要**具体**，不要宽泛
- 包含**具体触发词**

---

## 按质量分类的示例

### 优秀

```
从 PDF 文件中提取文本和表格，填写表单，合并文档。当处理 PDF 文件或用户提到 PDF、表单或文档提取时触发。
```
原因：具体的能力，具体的触发条件，没有歧义。

```
针对棘手问题的规范诊断循环：复现 -> 最小化 -> 假设 -> 检测 -> 修复 -> 回归测试。当用户说"诊断这个问题"/"调试这个问题"、报告 bug、说某物坏了/抛错/失败了，或描述性能回归时触发。
```
原因：清晰的过程摘要 + 详尽的触发条件列表。

```
关于计划或设计的每个方面无情地访谈用户，直到决策树的每个分支都被解决。当用户想要压力测试计划、被拷问设计，或提到"拷问我"时触发。
```
原因：生动的能力描述 + 自然语言触发条件。

### 良好

```
超压缩通信模式。通过丢弃填充词同时保持完整技术准确性，将 token 使用量减少约 75%。当用户说"穴居人模式"、"像穴居人一样说话"、"使用穴居人"、"少用点 token"、"简洁点"，或调用 /caveman 时触发。
```
原因：量化的收益 + 详尽的触发短语。

### 差

```
帮助处理文档。
```
原因：无法与其他文档技能区分。没有触发条件。

```
一个用于处理 GitHub 的技能。
```
原因：太宽泛 —— GitHub 的哪个方面？Issues？PR？Actions？

```
这个技能允许 Agent 执行与项目管理相关的各种任务，包括但不限于创建问题、管理待办事项，以及跟踪软件开发生命周期的多个维度上的进展。
```
原因：冗长，没有触发条件，使用第一人称，模糊的能力。

## 触发条件设计

好的触发条件覆盖用户可能请求能力的多种方式：

### 直接调用
- `/command-name`
- "使用 {技能名称}"

### 基于意图
- "想要 {行动}"
- "需要 {行动}"

### 基于关键词
- 提到 "{term1}" 或 "{term2}"
- 说 "{phrase1}" / "{phrase2}"

### 基于上下文
- "当处理 {文件类型} 时"
- "当用户 {情况} 时"

### 触发条件集合示例

```
当用户说"诊断这个问题"/"调试这个问题"、报告 bug、说某物坏了/抛错/失败了，或描述性能回归时触发。
```

这覆盖了：
- 直接："诊断这个问题"、"调试这个问题"
- 基于动词："报告 bug"
- 基于形容词："坏了"、"抛错"、"失败了"
- 基于名词："性能回归"

---

## 编写过程

1. **先写正文** —— 你需要知道技能实际做什么
2. **提取核心能力** —— 一句话，它解决什么问题？
3. **列出用户可能请求的 5-10 种方式** —— 自然语言变体
4. **组合成格式** —— 能力 + "当……时使用" + 触发条件
5. **修剪到 1024 字符** —— 砍掉填充词，保留 specifics
6. ** mentally 测试** —— "如果用户说 X，这个描述会匹配吗？"

---

## 常见错误

| 错误 | 修复 |
|------|-----|
| 太宽泛 | 添加具体的文件类型、动作或关键词 |
| 太窄 | 包含用户可能说的同义短语 |
| 没有触发条件 | 始终以"当……时使用"结尾 |
| 第一人称 | 使用第三人称（"该技能执行 X"）|
| 过去时 | 使用现在时 |
| 没有上下文的行话 | 在术语前添加简短的能力陈述 |
| 超过 1024 字符 | 修剪填充词，保留具体触发条件 |
