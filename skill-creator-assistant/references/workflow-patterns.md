# 工作流设计模式

如何在技能中构建逐步流程。

---

## 反馈循环原则

最重要的模式：**每个产出东西的技能也应该验证它**。

没有验证，Agent 就是在盲飞。有了验证，Agent 可以迭代走向正确。

构建反馈循环的方法：

| 方法 | 最适合 | 速度 |
|------|--------|------|
| 失败测试 | 代码变更 | 快 |
| 带固定输入的脚本 | 数据转换 | 快 |
| 对运行中的服务的 HTTP 调用 | API 变更 | 中等 |
| 与已知良好输出对比 | 内容生成 | 快 |
| 人工检查循环 | 主观质量 | 慢 |

**经验法则**：如果你的技能没有反馈机制，添加一个，或记录用户应如何验证输出。

---

## 渐进检查点

将长工作流分解为有检查点的阶段：

```
阶段 1 — 设置
- [ ] 步骤一
- [ ] 步骤二

在 [条件] 之前不要继续。

阶段 2 — 执行
- [ ] 步骤三
- [ ] 步骤四
```

为什么：如果出了问题，检查点让问题出在哪里变得显而易见。没有检查点，Agent 可能直到最后才注意到失败。

**关键短语**：
- "在……之前不要继续"
- "停下来明确说明如果……"
- "在继续之前确认：……"
- "在声明完成之前必须：……"

---

## 逃生舱口

每个工作流都需要一个出错时的出路：

```
### 当你无法 {期望的动作} 时

停下来明确说明。列出你尝试过的方法。向用户请求：
(a) {替代资源}，(b) {备选方案}，或 (c) {权限}。

不要带着假设继续。
```

没有逃生舱口，Agent 会幻觉出进展或陷入循环。

---

## 惰性初始化

只在首次需要时创建资源，不要预先创建：

```
如果没有 CONTEXT.md 存在，在第一个术语被确定时创建一个。
如果没有 docs/adr/ 存在，在第一个 ADR 需要时创建它。
```

这避免了在还不需要该资源的项目中产生混乱。它也使技能在任何项目状态下都能工作 —— 无论是全新项目还是现有项目。

---

## 实时文档更新

当技能产出决策或术语时，立即捕获它们：

```
当术语被确定时，立即更新 CONTEXT.md。不要批量处理 —— 发生时即捕获。
```

这防止了工作的丢失，并随时间积累项目上下文。技能使用得越多，效果越好。

**ADR 的决策标准**（只有在全部为真时才创建）：
1. 难以撤销
2. 没有上下文会令人惊讶
3. 真实权衡的结果

---

## 一次一个交互

对于访谈或询问用户的技能：

```
一次只问一个问题，等待每个问题的反馈后再继续。
```

为什么：多问题抛出让用户不知所措。一次一个问题让他们深入思考每个答案。它也允许 Agent 根据回答调整后续问题。

---

## 优雅降级

即使在不满足最优条件时，技能也应工作：

**硬依赖**：没有它技能无法功能。包含显式指向设置技能的指针："……应该已经提供给你了 —— 如果没有就运行 /setup-X。"

**软依赖**：有它技能工作得更好，但没有也能功能。模糊引用："如果可用，使用项目的领域词汇表。"

这种分离使软依赖技能保持轻量 token，并避免在不必要的地方机械地复制设置指针。

---

## 会话恢复

对于多会话工作流：

```
如果项目上存在先前的注释，阅读它们，检查自上次会话以来是否有新活动，
并在继续前展示更新后的画面。不要重复已解决的问题。
```

关键元素：
- 行动前阅读先前状态
- 检测自上次会话以来发生了什么变化
- 展示增量，不是完整回顾
- 保持上下文连续性

---

## 生成内容的免责声明

当技能代表用户发布内容（issues、评论等）时：

```
在此技能执行期间发布的每个评论或 issue 必须以以下声明开头：

> *这是在 {skill-name} 期间由 AI 生成的。*
```

这为人类读者设定了期望并创建了问责制。

---

## 一致术语强制

定义技能的词汇并精确使用：

```
在每个建议中精确使用这些术语。一致的语言是重点。

- **模块** —— 定义
- **接口** —— 定义
- **接缝** —— 定义（使用这个，不是"边界"）
```

这训练 Agent 使用一致的语言，从而：
- 减少 token 使用（更短的术语）
- 改善代码库导航（一致的命名）
- 防止歧义（一个术语，一个含义）
