# 技能模板

常见技能类型的可复用模板。选择最接近的模板并适配。

---

## 模板 A：诊断/调试技能

用于通过系统消除法调查问题的技能。

```md
---
name: {diagnostic-skill}
description: >
  系统性 {domain} 诊断，通过构建反馈循环和假设测试。当用户报告
  {problem-type}、说"诊断这个问题"/"调试这个问题"，或描述
  {symptom-pattern} 时触发。
---

# {诊断技能名称}

针对棘手 {problem-type} 的规范诊断方法。仅在明确有理由时跳过阶段。

## 阶段 1 — 构建反馈循环

**这就是技能本身。** 其余都是机械操作。如果你有一个快速、确定性的
通过/失败信号，你就能找到原因。

在这里投入不成比例的努力。**要激进。要有创意。**

构建反馈循环的方法（按此顺序尝试）：

1. **失败测试** — 在触及问题的接缝处。
2. **脚本调用** — 带固定输入。
3. **重放捕获的跟踪记录。**
4. **最小化测试台。** — 系统的子集，能复现问题路径。
5. **差异循环。** — 运行旧版 vs 新版，对比输出。

### 迭代优化循环

- 能让它更快吗？
- 能让信号更清晰吗？
- 能让它更确定性吗？

### 当无法构建循环时

停下来明确说明。列出你尝试过的方法。向用户请求：
(a) 复现环境的访问权限，(b) 捕获的工件，或 (c) 添加监测的权限。
**不要在循环准备好之前继续到阶段 2。**

## 阶段 2 — 复现

运行循环。确认：

- [ ] 循环产生了**用户**描述的失败
- [ ] 失败在多次运行中可复现
- [ ] 确切的症状已被捕获，供后续验证

在复现问题之前不要继续。

## 阶段 3 — 假设

在测试之前生成 **3-5 个排序的假设**。每个假设必须**可证伪**：

> "如果 <X> 是原因，那么 <改变 Y> 会让问题消失。"

在测试之前向用户展示排序列表。他们通常拥有能立即重新排序的领域知识。

## 阶段 4 — 检测

每次探测都映射到一个具体的预测。一次只改变一个变量。

1. **直接检查** — 如果环境支持。
2. **针对性日志** — 在假设边界处。
3. 绝不"到处打日志然后 grep"。

## 阶段 5 — 修复 + 回归测试

在修复**之前**写回归测试 — 但只在**正确的接缝**处。

正确的接缝能测试在调用点发生的**真实问题模式**。如果没有正确的接缝，
这本身就是发现：标记架构缺口。

## 阶段 6 — 清理

- [ ] 原始复现不再复现
- [ ] 回归测试通过（或接缝缺失已记录）
- [ ] 所有调试检测已移除
- [ ] 临时原型已删除
```

## 模板 B：访谈/对齐技能

用于需要在行动前与用户深度对齐的技能。

```md
---
name: {interview-skill}
description: >
  无情地访谈用户关于计划或设计的每个方面，直到决策树的每个分支
  都被解决。当用户想要压力测试计划、被拷问设计，或提到
  "拷问我"/" grill me "时触发。
---

关于这个计划的每个方面无情地访谈我，直到我们达成共同理解。
沿着设计树的分支深度优先走，逐个解决决策之间的依赖关系。
对每个问题，给出你的推荐答案。

一次只问一个问题，等待每个问题的反馈后再继续。

如果问题可以通过探索 {relevant-source} 来回答，那就去探索。

## 领域意识

在探索过程中，同时查找：

- 现有文档和约定
- 限制当前选项的先前决策
- 需要精确定义的术语

## 会话期间

### 挑战现有语言

当用户的术语与已建立的词汇冲突时，立即指出。
"你的词汇表将'X'定义为 Y，但你似乎指的是 Z —— 到底是哪个？"

### 锐化模糊语言

当用户使用模糊或超载的术语时，提出精确的规范术语。

### 讨论具体场景

用具体的边缘案例场景来压力测试领域关系。

### 实时更新文档

当术语被确定时，立即捕获。不要批量处理 —— 发生时即捕获。
```

## 模板 C：工作流/状态机技能

用于项目通过定义的状态流转的技能。

```md
---
name: {workflow-skill}
description: >
  通过 {roles/stages} 的状态机移动 {items}。当用户想要
  {action}、管理 {item} 工作流，或说 {trigger-phrase} 时触发。
---

# {工作流名称}

通过 {roles} 的小型状态机移动 {items}。

## 角色/状态

- `{state-a}` — 描述
- `{state-b}` — 描述
- `{state-c}` — 描述

状态流转：正常流程是 A -> B -> C。标记异常流转并在继续前询问。

## 调用方式

用自然语言解释用户的请求。示例：

- "给我看需要我关注的所有东西"
- "看看 #{id}"
- "把 #{id} 移到 {state}"

## 处理特定项目的流程

1. **收集上下文。** 阅读完整项目。解析先前的注释以避免重复已解决的问题。
   探索相关上下文。
2. **推荐。** 说明你的推荐及理由。等待指示。
3. **根据结果行动：**
   - `{state-ready}` — 发布简短评论
   - `{state-blocked}` — 发布带具体问题的注释
   - `{state-closed}` — 礼貌解释，然后关闭

## 恢复之前的会话

如果存在先前的注释，阅读它们，检查是否有新活动，并展示更新后的
画面再继续。不要重复已解决的问题。
```

## 模板 D：转换/生成技能

用于将输入 A 转换为输出 B 的技能。

```md
---
name: {transform-skill}
description: >
  将 {input-type} 转换为 {output-type}。当用户想要从 {input} 创建
  {output}，或提到 {trigger-phrase} 时触发。
---

# {转换名称}

使用当前上下文将 {input} 转换为 {output}。

## 流程

1. **理解当前状态。** 如果需要，探索相关上下文。
2. **草拟结构。** 确定主要部分/模块。
3. **应用模板。** 使用下面的格式。

## 输出模板

```
## 问题陈述
{描述}

## 解决方案
{描述}

## 细节
{编号列表}

## 范围外
{描述}

## 备注
{其他内容}
```

## 规则

- 不要包含具体的文件路径或代码片段（它们会很快过时）
- 不要访谈用户 —— 综合已知信息
- 应用适当的标签以便输出进入正常工作流
```

## 模板 E：通信模式技能

用于改变 Agent 通信方式的技能。

```md
---
name: {mode-skill}
description: >
  {通信风格}模式。通过 {机制} 减少 token 使用量。当用户说
  "{trigger-phrase}"、"{alt-trigger}"，或调用 /{command} 时触发。
---

以 {风格描述} 的方式回应。

## 持久性

一旦触发，**每个回应都保持激活**。多轮后不会自动恢复。
只有当用户说 "{stop-phrase}" 或 "{alt-stop-phrase}" 时才关闭。

## 规则

丢弃：{要移除的元素列表}
保留：{要保持精确的元素列表}

模式：`{期望的通信模式}`
不要：`{不期望的模式示例}`
要：`{期望的模式示例}`

## 例外

在以下情况临时丢弃 {模式}：{例外列表}。完成后恢复。
```
